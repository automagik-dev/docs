# Genie is the orchestration layer for multi-agent development

<Note>
  **Research Preview** — Genie is experimental software under active development. APIs will change, things will break, and we're building in public. That's the point. Join [Discord](https://discord.gg/automagik) to shape what this becomes.
</Note>

## The problem isn't AI — it's context collapse

AI agents are table stakes. The problem isn't whether they can code — it's whether they can code **together**.

You've felt this. You have five agents open. One is refactoring the auth module. Another is building the API endpoint that depends on it. A third is writing tests against an interface that changed twenty minutes ago. None of them know what the others are doing.

So you re-explain. You copy context between tabs. You paste the same file three times into three different conversations. And when the third agent hallucinates an import path because it never saw the refactor — you fix it yourself, defeated, wondering why you're the bottleneck in a system that was supposed to remove bottlenecks.

**This is context collapse.** Not a failure of any single agent, but the absence of anything holding them together. Five brilliant musicians, no conductor, no sheet music, no shared key signature.

## Genie is the conductor

Genie doesn't replace your agents — it orchestrates them. It's the layer between your intent and their execution that handles everything agents can't handle alone:

- **Shared context** — every agent reads from the same wish, the same scope, the same acceptance criteria
- **Coordinated execution** — parallel work across isolated worktrees, no merge conflicts during development
- **Structured lifecycle** — brainstorm, plan, build, review, ship — not "figure it out"
- **Cross-agent messaging** — agents talk to each other, not through you

You describe what you want. Genie breaks it into parallelizable work units, assigns agents, manages their state, and assembles the result into a pull request. You review and merge. That's it.

```
/brainstorm → /wish → /work → /review → ship
```

## Bring Your Own Agent

Genie is agent-agnostic. It's an orchestration protocol, not a wrapper around one vendor. Use whatever agent fits the job — or mix them on the same team.

| Agent | Setup | Best For |
|-------|-------|----------|
| **Claude Code** | Works out of the box | Deepest integration — native plugin, team support, hooks |
| **Codex** | `genie setup --codex` | OpenAI's agent — full transcript and team support |
| **Open Claw** | `genie setup --open-claw` | Open-source option, community-maintained adapter |
| **Custom (BYOA)** | `genie setup --custom` | Any CLI agent with stdout/stderr — bring your own |

```bash
# Mix providers on the same team — fast engineers, thorough reviewer
genie spawn engineer --provider claude --model sonnet
genie spawn engineer --provider codex --model codex-mini
genie spawn reviewer --provider claude --model opus
```

Your agents are interchangeable. Switch mid-project, mix on a team, or write your own adapter. Genie doesn't care which agent you use — it cares that the work ships.

</Tip>

## 8 Core Capabilities

| Capability | What It Does | Key Feature |
|------------|-------------|-------------|
| **Wishes** | Structured plans with scope, criteria, and execution groups | Parallelizable work units with dependency ordering |
| **Skills** | 14 built-in slash commands — composable prompt programs | `/brainstorm` through `/dream` — each encodes a best practice |
| **Teams** | Multi-agent squads with shared state | Isolated worktrees per team, zero conflicts during dev |
| **Agents** | tmux-based workers with identity, roles, and heartbeats | Auto-spawn, crash recovery, idle suspension |
| **BYOA** | Provider-agnostic agent orchestration | Claude, Codex, Open Claw, or any custom CLI agent |
| **Worktrees** | Git worktree isolation per team | Parallel branches, automatic cleanup, shared `.genie/` state |
| **Messaging** | Cross-agent communication system | Mailbox delivery, team chat, broadcast, structured routing |
| **Hooks** | Event-driven automation on every tool call | Auto-spawn, identity injection, NATS integration |

## How it works

Every feature follows the same lifecycle — from vague idea to merged pull request:

<Steps>
  <Step title="Brainstorm">
    Explore the idea. Identify constraints, scope, and design decisions. The `/brainstorm` skill guides you through this interactively until the idea is crisp enough to plan.
  </Step>
  <Step title="Wish">
    Convert the brainstorm into a structured plan — `WISH.md` — with scope, acceptance criteria, and parallelizable execution groups. This is the contract between you and the agents.
  </Step>
  <Step title="Work">
    Genie dispatches agents per execution group. Independent groups run in parallel across isolated worktrees. Each agent knows its scope, its criteria, and how to validate its own output.
  </Step>
  <Step title="Review">
    A reviewer agent validates all work against the wish criteria. The verdict is **SHIP** or **FIX-FIRST**. Fix loops run automatically — up to the configured retry limit.
  </Step>
  <Step title="Ship">
    A pull request is created, targeting `dev`. You review and merge. The agents did the work; you made the decision.
  </Step>
</Steps>

## Desktop is coming

Genie is CLI-first today — built for agents and developers who live in the terminal. But we're building a desktop application for Q2 2026 that brings the same orchestration to a visual interface: wish boards, agent dashboards, real-time team monitoring.

The CLI isn't going anywhere. The desktop is another surface for the same system.
