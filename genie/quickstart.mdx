# Quickstart

Get Genie running and ship your first feature — with whatever AI agent you prefer.

<Note>
  **Research Preview** — Genie is experimental. Things will break, APIs will change, and we want your feedback.
  Join [Discord](https://discord.gg/automagik) to tell us what's working and what isn't.
</Note>

<Steps>
  <Step title="Install Genie">
    Run the bootstrap script. It installs the CLI, sets up plugins, and configures git hooks.

    ```bash
    curl -fsSL https://raw.githubusercontent.com/automagik-dev/genie/main/install.sh | bash
    ```

    Verify the install:

    ```bash
    genie doctor
    ```

    Expected output:
    ```
    ✓ genie CLI found
    ✓ Agent detected
    ✓ tmux available
    ✓ Plugin symlink active
    ✓ Git hooks configured
    ```
  </Step>

  <Step title="Choose Your Agent">
    Genie is an orchestration layer, not an agent. It works **with** your agent — whichever one you prefer. We don't care which one you marry; we care that the wedding runs on time.

    | Agent | Setup | Notes |
    |-------|-------|-------|
    | **Claude Code** | Installed by default | Recommended — deepest integration, native plugin support |
    | **Codex** | `genie setup --codex` | OpenAI's agent. Full transcript + team support |
    | **Open Claw** | `genie setup --open-claw` | Open-source option. Community-maintained adapter |
    | **Custom (BYOA)** | `genie setup --custom` | Bring Your Own Agent — any CLI agent with stdout/stderr |

    Claude Code ships out of the box. For other agents, run the setup command above and Genie configures the adapter automatically.

    <Tip>
      **Bring Your Own Agent** — Genie's adapter system means you're never locked in.
      Switch agents mid-project, mix agents on the same team, or write your own adapter.
      See the [adapter docs](/genie/architecture) for details.
    </Tip>
  </Step>

  <Step title="Paste Into Your Agent">
    Open your agent in your project directory and paste:

    ```
    You are now a Genie-powered agent. Run `genie` to initialize,
    then use /wizard for guided onboarding.
    ```

    Genie detects a fresh project and scaffolds everything:

    <Frame>
      <CodeGroup>
```bash Scaffold Flow
$ genie
🔍 Scanning project...

⚠ No AGENTS.md found — this looks like a fresh project.

Scaffolding your agent workspace...
  Creating AGENTS.md...        ✓
  Creating SOUL.md...          ✓
  Creating .genie/wishes/...   ✓
  Creating .genie/state/...    ✓
  Configuring git hooks...     ✓

✨ Project scaffolded successfully!

Your agent workspace is ready:
  AGENTS.md    → agent directory and roles
  SOUL.md      → agent identity and principles
  .genie/      → wishes, state, mailbox, chat

Next: run /wizard for guided onboarding.
```
      </CodeGroup>
    </Frame>
  </Step>

  <Step title="Brainstorm Your First Idea">
    Use the `/brainstorm` skill to explore an idea:

    ```
    /brainstorm Add a dark mode toggle to the settings page
    ```

    The brainstorm skill guides you through:
    - Scope definition — what's in, what's out
    - Technical constraints — dependencies, breaking changes
    - Design decisions — approach, tradeoffs
    - Wish readiness — when the idea is crisp enough to plan
  </Step>

  <Step title="Create a Wish">
    When the brainstorm is ready, convert it to a structured plan:

    ```
    /wish
    ```

    This produces a `WISH.md` with:
    - Summary and scope
    - Acceptance criteria (checkboxes)
    - Execution groups (parallelizable work units)
    - Success validation commands
  </Step>

  <Step title="Execute and Ship">
    Run the wish — Genie dispatches agents per execution group:

    ```
    /work
    ```

    When all groups complete, review the result:

    ```
    /review
    ```

    If the review returns **SHIP**, a PR is created. If **FIX-FIRST**, the fix loop runs automatically.
  </Step>
</Steps>

## What Just Happened?

You went from an idea to a pull request using five slash commands — and your agent did the actual work. Here's the full pipeline:

<Frame>
  <CodeGroup>
```bash /brainstorm
$ /brainstorm Add a dark mode toggle to the settings page

🧠 Brainstorming "dark-mode-toggle"...

Exploring scope...
  ✓ Dark mode CSS variables
  ✓ Toggle component in settings
  ✓ Persistent preference via localStorage

Identifying constraints...
  ✓ Must avoid FOUC on page load
  ✓ Existing component tests must pass

Design crystallized → ready for /wish
  Draft: .genie/brainstorms/dark-mode-toggle/DESIGN.md
```

```bash /wish
$ /wish

📋 Converting brainstorm to wish...
  Source: .genie/brainstorms/dark-mode-toggle/DESIGN.md

Writing WISH.md...
  ✓ Summary and scope
  ✓ 4 success criteria
  ✓ 2 execution groups (1 parallel, 1 sequential)

Wish created: .genie/wishes/dark-mode-toggle/WISH.md
Status: APPROVED — ready for /work
```

```bash /work
$ /work

🔨 Executing wish "dark-mode-toggle"...

Wave 1 (parallel):
  ✅ Group 1: CSS Variables         engineer  done   14:30
  ✅ Group 2: Toggle Component      engineer  done   14:32

All groups complete. Running validation...
  ✓ bun test — 42 passed, 0 failed
  ✓ bun run typecheck — no errors

Ready for /review
```

```bash /review
$ /review

🔍 Reviewing wish "dark-mode-toggle"...

Checking criteria:
  ✅ Toggle renders in settings page
  ✅ CSS variables switch between light/dark
  ✅ Preference persists across sessions
  ✅ No FOUC on page load
  ✅ All existing tests pass

Verdict: SHIP ✨

Creating pull request...
  PR #47: "feat: add dark mode toggle to settings"
  Branch: feat/dark-mode-toggle → dev
  URL: https://github.com/myapp/myapp/pull/47
```
  </CodeGroup>
</Frame>

| Step | Skill | Output |
|------|-------|--------|
| Explore | `/brainstorm` | Design document |
| Plan | `/wish` | WISH.md with criteria |
| Build | `/work` | Code changes across execution groups |
| Validate | `/review` | SHIP or FIX-FIRST verdict |
| Ship | PR created | Ready for human merge |
